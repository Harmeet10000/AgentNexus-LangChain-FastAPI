01 / Architecture Foundations

Structure before prompting

The most important insight from all three projects: how you architect context flow, isolation, and memory matters more than prompt quality. A well-structured agent with average prompts outperforms a poorly structured agent with excellent prompts.

The "Give the Agent a Computer" Principle

Anthropic's core finding: rather than building a chatbot with tools bolted on, give the model the same tools a human operator would have — a filesystem, a shell, the ability to run code. This is what makes the agent general-purpose.

Anthropic Principle

The key design principle is to give your agents a computer, allowing them to work like humans do. File system tools (ls, read_file, write_file, edit_file) allow agents to offload large context to storage, preventing context window overflow.

def execute_tool(tool_name: str, tool_input: dict) -> str:
    if tool_name == "bash":
        result = subprocess.run(
            tool_input["command"], shell=True,
            capture_output=True, text=True, timeout=30
        )
        return result.stdout + result.stderr

    elif tool_name == "read_file":
        with open(tool_input["path"]) as f:
            return f.read()

    elif tool_name == "write_file":
        with open(tool_input["path"], "w") as f:
            f.write(tool_input["content"])
        return f"Written to {tool_input['path']}"

Separation of Concerns: Read vs Write Agents

OpenCode's most important architectural decision: a Plan agent (read-only, analysis) and a Build agent (full tool access). These are distinct cognitive modes with different risk profiles. Build this separation into your architecture, not as a toggle.

🔍

Plan / Analyze Agent

Read-only tool access. ls, read_file, grep, git log. No writes, no shell mutations. Use for code review, research, summarization. Safer to run with lower human oversight.

⚙️

Build / Execute Agent

Full tool access including write_file, bash with mutations, package installs. Requires explicit permission gates. Should always have iteration limits enabled.

from enum import Enum
from dataclasses import dataclass, field
from typing import List, Literal

class Permission(Enum):
    ALLOW = "allow"
    ASK   = "ask"     # prompt user before running
    DENY  = "deny"

@dataclass
class AgentConfig:
    name: str
    mode: Literal["plan", "build"]
    max_iterations: int = 20      # hard limit — ALWAYS set this
    tool_permissions: dict = field(default_factory=dict)

    def can_run_tool(self, tool_name: str) -> Permission:
        return self.tool_permissions.get(tool_name, Permission.DENY)

# Plan agent: read-only, safe to run autonomously
PLAN_AGENT = AgentConfig(
    name="plan",
    mode="plan",
    max_iterations=15,
    tool_permissions={
        "bash":       Permission.ASK,    # only read-safe bash
        "read_file":  Permission.ALLOW,
        "write_file": Permission.DENY,   # no writes
        "glob":       Permission.ALLOW,
    }
)

# Build agent: full access, respect iteration limits
BUILD_AGENT = AgentConfig(
    name="build",
    mode="build",
    max_iterations=50,
    tool_permissions={
        "bash":       Permission.ALLOW,
        "read_file":  Permission.ALLOW,
        "write_file": Permission.ALLOW,

02 / The Agent Loop

Gather → Act → Verify

Anthropic's three-phase model for every agent session. The loop isn't just a pattern — it's the mechanism through which an agent corrects itself.

🔎
Gather Context
Files, search, memory
→
⚡
Take Action
Tools, code, bash
→
✓
Verify Work
Lint, test, screenshot
→
🔁
Repeat
Until done or limit hit
agent-loop.py
Python 
def run_agent_loop(
    system_prompt: str,
    initial_message: str,
    config: AgentConfig,
    tools: list
) -> str:
    """
    The core agent loop. Runs until:
    1. Model returns stop_reason='end_turn' (no more tool calls)
    2. max_iterations is hit (safety limit)
    """
    messages = [{"role": "user", "content": initial_message}]
    iteration = 0

    while iteration < config.max_iterations:
        iteration += 1
        print(f"[loop] iteration {iteration}/{config.max_iterations}")

        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=8096,
            system=system_prompt,
            tools=tools,
            messages=messages
        )

        # Append assistant turn to history
        messages.append({"role": "assistant", "content": response.content})

        # No tool calls → agent is done
        if response.stop_reason == "end_turn":
            print("[loop] agent finished naturally")
            break

        # Process tool calls
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue

            permission = config.can_run_tool(block.name)

            if permission == Permission.DENY:
                result = f"DENIED: tool '{block.name}' is not permitted in {config.mode} mode"
            elif permission == Permission.ASK:
                result = ask_user_permission(block.name, block.input)
            else:
                result = execute_tool(block.name, block.input)

            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result
            })

        messages.append({"role": "user", "content": tool_results})

    else:
        print(f"[loop] HIT MAX ITERATIONS ({config.max_iterations}). Forcing stop.")

    # Extract final text response
    for block in reversed(messages[-1]["content"]
                        if isinstance(messages[-1]["content"], list)
                        else []):
        if hasattr(block, "text"):
            return block.text
    return ""
03 / Context Engineering

Context is the scarcest resource

Every architectural decision in OpenCode, DeepAgents, and Anthropic's SDK is fundamentally about managing context. The context window is not working memory — it's a bottleneck. Design your context budget before writing a single tool.

Key Insight

The folder and file structure of an agent becomes a form of context engineering. The filesystem is memory the model can selectively load — don't cram everything into the context window upfront.

Agentic Search vs Semantic Search

Anthropic's recommendation: start with agentic search (grep, find, tail) and only add semantic/vector search if you need faster lookup. Agentic search is more accurate, more transparent, and easier to maintain — the tradeoff is latency.

context-management.py
Python 
import os, subprocess
from pathlib import Path

class AgentMemory:
    """
    Filesystem-as-memory pattern.
    The agent decides WHAT to load, not the harness.
    """
    def __init__(self, workspace: str):
        self.workspace = Path(workspace)
        self.workspace.mkdir(parents=True, exist_ok=True)

    def agentic_search(self, query: str, path: str = ".") -> str:
        """Let the agent grep for what it needs — don't preload everything."""
        cmd = f"grep -r --include='*.txt' --include='*.md' -l '{query}' {path}"
        result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
        return result.stdout

    def smart_read(self, file_path: str, max_lines: int = 500) -> str:
        """For large files, read head + tail rather than full content."""
        path = Path(file_path)
        if not path.exists():
            return f"File not found: {file_path}"

        lines = path.read_text().splitlines()

        if len(lines) <= max_lines:
            return path.read_text()

        # Return head + tail with truncation notice
        head = lines[:max_lines // 2]
        tail = lines[-(max_lines // 2):]
        skipped = len(lines) - max_lines
        return (
            "\n".join(head)
            + f"\n\n... [{skipped} lines omitted] ...\n\n"
            + "\n".join(tail)
        )

    def write_progress(self, entry: str) -> None:
        """Persistent progress log — the agent's cross-session memory."""
        progress_file = self.workspace / "agent-progress.md"
        with open(progress_file, "a") as f:
            from datetime import datetime
            timestamp = datetime.now().isoformat()
            f.write(f"\n## {timestamp}\n{entry}\n")

    def read_progress(self) -> str:
        progress_file = self.workspace / "agent-progress.md"
        if progress_file.exists():
            return progress_file.read_text()
        return "No progress recorded yet."

# Usage: inject into system prompt at session start
def build_context_prompt(memory: AgentMemory) -> str:
    return f"""
You have access to a workspace at {memory.workspace}.
On every session start:
1. Run `cat agent-progress.md` to get prior context
2. Use grep/find to locate relevant files — do NOT read everything upfront

4. Write progress updates after completing each subtask
"""

Context Compaction

When running long agent sessions, context windows fill up. Compaction is necessary but not sufficient — it summarizes tokens but doesn't preserve structured state. Always combine compaction with an explicit progress file.

context-compaction.py
Python 
def compact_conversation(messages: list, keep_last: int = 10) -> list:
    """
    When context grows too large, summarize early messages.
    This mirrors OpenCode's hidden 'compaction agent' pattern.
    """
    if len(messages) <= keep_last + 2:
        return messages

    old_messages = messages[:-(keep_last)]
    recent_messages = messages[-(keep_last):]

    # Use the model itself to summarize old context
    summary_response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # cheap model for infra tasks
        max_tokens=2000,
        messages=[
            *old_messages,
            {
                "role": "user",
                "content": """Summarize the work done so far in this conversation.
Include: what tasks were completed, what files were modified,
what decisions were made, and what the current state is.
Be concrete and detailed — this summary will be the only context
a new agent instance has about prior work."""
            }
        ]
    )

    summary_text = summary_response.content[0].text

    # Return compacted history: summary + recent messages
    return [
        {
            "role": "user",
            "content": f"[CONTEXT SUMMARY OF PRIOR WORK]\n{summary_text}"
        },
        {
            "role": "assistant",
            "content": "Understood. I have the summary of prior work. Continuing..."
        },
        *recent_messages
    ]

# Auto-compact when approaching token limit
def run_with_compaction(messages: list, token_limit: int = 150000) -> list:
    # Rough estimate: ~3.5 chars per token
    total_chars = sum(len(str(m)) for m in messages)
    estimated_tokens = total_chars / 3.5

    if estimated_tokens > token_limit * 0.8:  # compact at 80%
        print(f"[compaction] ~{estimated_tokens:.0f} tokens, compacting...")
        return compact_conversation(messages)

    return messages
04 / Long-Running Agent Harness

Initializer + Coding Agent Pattern

Anthropic's most important production finding: long-running agents need a two-part harness. Without it, agents either declare victory prematurely, or get stuck fixing half-implemented features from previous sessions.

Production Failure Modes

Two failure modes in the wild: (1) Agent one-shots the entire task, runs out of context mid-way, leaves environment broken. (2) Later agent instance sees partial progress and declares the job done. Both are solved by the initializer + coding agent harness.

The Initializer Agent

The first session uses a specialized prompt to set up structured scaffolding that all future agents will depend on. This runs once and never again.

initializer-agent.py
Python 
INITIALIZER_SYSTEM_PROMPT = """You are a project initializer agent. Your job is NOT to build
the project. Your job is to set up the environment so that future
coding agents can build it incrementally without confusion.

You MUST create these four files before doing anything else:

1. feature_list.json  — Full list of all required features as JSON.
   Every feature must have "passes": false initially.
   NEVER mark anything as true yet.

2. agent-progress.md — Empty progress log with just a header.

3. init.sh — Script to start the dev server and run a basic health check.

4. AGENTS.md — One-paragraph explanation of the tech stack and
   architecture decisions, so future agents understand the codebase.

After creating these files, make an initial git commit.
Do NOT start implementing any features.
"""

INITIALIZER_FEATURE_SCHEMA = """
{
  "features": [
    {
      "id": "F001",
      "category": "core",
      "description": "User can create a new item",
      "acceptance_criteria": [
        "Form appears when clicking Add button",
        "Item appears in list after submission",
        "Empty form clears after submit"
      ],
      "passes": false
    }
  ]
}
"""

def run_initializer(project_description: str, workspace: str) -> None:
    print("[init] Running initializer agent...")
    run_agent_loop(
        system_prompt=INITIALIZER_SYSTEM_PROMPT,
        initial_message=f"""Project: {project_description}

Workspace: {workspace}

Create the 4 required files and make the initial git commit.
Feature list schema to follow:\n{INITIALIZER_FEATURE_SCHEMA}
""",
        config=BUILD_AGENT,
        tools=AGENT_TOOLS
    )
    print("[init] Initialization complete.")

The Coding Agent (Incremental Loop)

coding-agent.py
Python 
CODING_AGENT_SYSTEM_PROMPT = """You are a coding agent working incrementally on a project.
Every session, you follow these exact steps in order:

STARTUP SEQUENCE (do these first, every time):
1. Run `pwd` to confirm your working directory
2. Read agent-progress.md to understand what was done before
3. Read feature_list.json to see what features remain
4. Run `bash init.sh` to start the dev server
5. Run a basic end-to-end test to confirm the app is not broken

WORK SEQUENCE:
6. Pick ONE feature with "passes": false — the highest priority one
7. Implement ONLY that feature — do not work on multiple at once
8. Test the feature as a human user would (browser automation if available)
9. Only set "passes": true after manually verifying it works end-to-end
   — It is UNACCEPTABLE to mark a feature passing without testing it

CLOSING SEQUENCE:
10. Commit your changes with a descriptive git message
11. Append a progress entry to agent-progress.md describing what you did,
    what was tricky, and what the current state of the app is
12. If there are failing tests or bugs, document them in the progress log

CRITICAL RULES:
- NEVER remove or edit feature acceptance criteria
- NEVER work on more than one feature per session
- NEVER declare the project complete unless ALL features pass
- JSON files are contracts — do not restructure them
"""

def run_coding_session(workspace: str, session_num: int) -> None:
    print(f"[session {session_num}] Starting coding agent...")
    run_agent_loop(
        system_prompt=CODING_AGENT_SYSTEM_PROMPT,
        initial_message=f"""You are in workspace: {workspace}
This is session #{session_num}.
Begin the startup sequence now.""",
        config=BUILD_AGENT,
        tools=AGENT_TOOLS
    )

def run_project(description: str, workspace: str, max_sessions: int = 20) -> None:
    run_initializer(description, workspace)  # session 0

    for session in range(1, max_sessions + 1):
        run_coding_session(workspace, session)

        # Check if all features pass
        import json
        features = json.loads((Path(workspace) / "feature_list.json").read_text())
        all_pass = all(f["passes"] for f in features["features"])

        if all_pass:
            print(f"[project] All features passing after session {session}!")
            break
    else:
        print(f"[project] Hit max sessions ({max_sessions}).")
Why JSON over Markdown?

Anthropic found that the model is less likely to inappropriately change or overwrite JSON files compared to Markdown files. For any file that is a contract (feature list, test specs), use JSON. Use Markdown only for prose logs where organic editing is expected.

05 / Sub-Agents & Parallelization

Context isolation, not just throughput

The reason to use sub-agents isn't performance — it's context hygiene. The main orchestrator stays coherent when it only receives final results, not every intermediate step from a deep research task.

🔀

Parallelization

Spin up multiple subagents simultaneously on independent subtasks. The orchestrator collects results when all complete. Useful for broad research across many sources.

🧹

Context Isolation

Each subagent uses its own context window and returns only the final cleaned output. Prevents the main agent's context from being polluted by intermediate reasoning noise.

subagents.py
Python 
import asyncio
from typing import List, TypedDict

class SubAgentResult(TypedDict):
    task: str
    result: str
    success: bool

async def run_subagent(
    task: str,
    system_prompt: str,
    tools: list,
    max_iterations: int = 10
) -> SubAgentResult:
    """
    Each subagent runs in its OWN context window.
    Orchestrator only sees the final output — not the intermediate steps.
    This is the key insight: context isolation, not just parallelism.
    """
    print(f"  [subagent] starting task: {task[:60]}...")

    # Subagent runs with its own isolated loop
    messages = [{"role": "user", "content": task}]
    config = AgentConfig(
        name="subagent", mode="plan",
        max_iterations=max_iterations
    )

    result = run_agent_loop(system_prompt, task, config, tools)

    return SubAgentResult(task=task, result=result, success=True)

async def orchestrate_parallel(
    tasks: List[str],
    subagent_system_prompt: str,
    tools: list
) -> List[SubAgentResult]:
    """
    Orchestrator pattern from LangChain DeepAgents / Anthropic SDK:
    - Spawn N subagents in parallel
    - Each gets isolated context
    - Collect only final results
    - Pass cleaned results to main agent
    """
    coroutines = [
        run_subagent(task, subagent_system_prompt, tools)
        for task in tasks
    ]
    results = await asyncio.gather(*coroutines, return_exceptions=True)

    # Filter failures
    clean_results = []
    for r in results:
        if isinstance(r, Exception):
            print(f"  [subagent] ERROR: {r}")
        else:
            clean_results.append(r)

    return clean_results

# Supervisor-researcher pattern (from open_deep_research)
RESEARCHER_PROMPT = """You are a focused research agent. You have one job:
research the specific topic assigned to you thoroughly.
Return ONLY a structured summary. Do NOT include your search queries,
intermediate steps, or failed attempts. Clean output only."""

SUPERVISOR_PROMPT = """You are a synthesis agent. You receive research
summaries from multiple specialist researchers. Your job is to:
1. Identify themes and contradictions across sources
2. Weight evidence by source quality
3. Produce a coherent unified analysis
You do NOT do primary research — only synthesis."""

async def deep_research(main_topic: str, subtopics: List[str]) -> str:
    # Phase 1: parallel research across subtopics
    research_results = await orchestrate_parallel(
        tasks=[f"Research: {t}" for t in subtopics],
        subagent_system_prompt=RESEARCHER_PROMPT,
        tools=AGENT_TOOLS
    )

    # Phase 2: supervisor synthesizes — only sees clean summaries
    summaries = "\n\n---\n\n".join(
        f"Research on '{r['task']}':\n{r['result']}"
        for r in research_results
    )

    synthesis = run_agent_loop(
        system_prompt=SUPERVISOR_PROMPT,
        initial_message=f"Topic: {main_topic}\n\nResearcher summaries:\n{summaries}",
        config=AgentConfig(name="supervisor", mode="plan", max_iterations=5),
        tools=[]  # supervisor only synthesizes — no research tools
    )

    return synthesis
06 / Tool Design Principles

Every tool must justify its existence

Tools are prominent in the context window — they compete with your instructions for the model's attention. Poorly designed tools create ambiguity, overlap, and wasted tokens on every single call.

Anthropic's Tool Principle

Tools should be self-contained, non-overlapping, and purpose-specific. Be conscious about tool design to maximize context efficiency. Your tools should be the primary, most frequent actions you want the agent to take — not a dumping ground for every possible capability.

Deterministic Tools > LLM Reasoning (when both work)

OpenCode integrates LSP. DeepAgents uses a real filesystem backend. Both reach for battle-tested deterministic systems where they exist. The LLM handles judgment; deterministic tools handle precision.

deterministic-tools.py
Python 
# Instead of asking the LLM "is this code valid?",
# run the actual linter and feed results back

def lint_and_typecheck(file_path: str) -> str:
    """
    Deterministic feedback > LLM guessing.
    TypeScript gives multiple error layers:
    - Syntax errors (tsc)
    - Type errors (tsc strict)
    - Style errors (eslint)
    Anthropic: TypeScript > JavaScript because of this extra feedback.
    """
    results = []

    # Type checking
    tsc = subprocess.run(
        ["npx", "tsc", "--noEmit", file_path],
        capture_output=True, text=True
    )
    if tsc.returncode != 0:
        results.append(f"TYPE ERRORS:\n{tsc.stdout}")

    # Linting
    eslint = subprocess.run(
        ["npx", "eslint", "--format=compact", file_path],
        capture_output=True, text=True
    )
    if eslint.returncode != 0:
        results.append(f"LINT ERRORS:\n{eslint.stdout}")

    if not results:
        return "✓ No errors found"

    return "\n\n".join(results)

# Email validation tool — deterministic rule, not LLM judgment
def validate_before_send(email_to: str, content: str) -> dict:
    import re
    errors, warnings = [], []

    if not re.match(r'^[\w.+-]+@[\w-]+\.[a-z]{2,}$', email_to):
        errors.append(f"Invalid email address: {email_to}")

    if len(content) < 10:
        errors.append("Content too short — likely malformed")

    if len(content) > 50000:
        warnings.append("Very long email — consider summarizing")

    return {"valid": len(errors) == 0, "errors": errors, "warnings": warnings}
07 / Explicit Planning

The TODO pattern

LangChain DeepAgents' most important insight: forcing the agent to call a TODO tool before execution dramatically improves long-term coherence. This isn't just scaffolding — it externalizes the agent's reasoning into a persistent, revisitable structure.

Why This Works

A plan that lives in a file is observable, correctable, and persistent across context windows. A plan that lives only in the model's reasoning trace is invisible, ephemeral, and lost on compaction. External plans are better plans.

08 / Verification & Self-Evaluation

Three layers of verification

Anthropic identifies three verification modes of increasing fuzziness. Always start with rules-based — it's the most reliable and cheapest. Add visual and LLM-as-judge only when rules-based coverage isn't sufficient.

Layer 1: Rules-Based (Deterministic)

verification-rules.py

from dataclasses import dataclass
from typing import Callable, List

@dataclass
class VerificationRule:
    name: str
    check: Callable[[str], bool]
    error_message: str
    is_blocking: bool = True

class AgentVerifier:
    """
    Rules-based verification for agent outputs.
    The more specific the rules, the better the feedback loop.
    Best form of feedback: clearly defined rules + explanation of what failed.
    """
    def __init__(self, rules: List[VerificationRule]):
        self.rules = rules

    def verify(self, output: str) -> dict:
        errors, warnings = [], []

        for rule in self.rules:
            try:
                passed = rule.check(output)
            except Exception as e:
                passed = False
                rule.error_message = f"{rule.error_message} (exception: {e})"

            if not passed:
                (errors if rule.is_blocking else warnings).append(
                    f"RULE '{rule.name}': {rule.error_message}"
                )

        return {
            "passed": len(errors) == 0,
            "errors": errors,
            "warnings": warnings,
            "feedback": "\n".join(errors + warnings) or "✓ All rules passed"
        }

# Example: email agent output verifier

def llm_judge(
    output: str,
    rubric: str,
    reference_examples: list = None
) -> dict:
    """
    LLM-as-judge pattern. Use ONLY for fuzzy criteria:
    - Tone / style alignment
    - Coherence / logical flow
    - Format consistency
    NOT a replacement for rules-based checks.
    Anthropic: this has heavy latency tradeoffs.
    Use a cheap/fast model (Haiku) to minimize cost.
    """
    examples_text = ""
    if reference_examples:
        examples_text = "Reference examples:\n" + "\n---\n".join(reference_examples)

    judge_prompt = f"""You are evaluating agent output against a quality rubric.

RUBRIC:
{rubric}

{examples_text}

OUTPUT TO EVALUATE:
{output}

Respond with ONLY valid JSON:
{{
  "score": <1-5>,
  "passed": = 4>,
  "issues": ["list of specific problems"],
  "suggestion": "one specific improvement"
}}"""

    response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # cheap model for judging
        max_tokens=500,
        messages=[{"role": "user", "content": judge_prompt}]
    )

    try:
        return json.loads(response.content[0].text)
    except:
        return {"score": 0, "passed": False, "issues": ["Judge failed to parse"]}

# Combined verification pipeline
def full_verification_pipeline(output: str, context: dict) -> dict:
    # Step 1: fast, deterministic rules (always run)
    rules_result = email_verifier.verify(output)
    if not rules_result["passed"]:
        return rules_result  # fail fast, don't waste LLM calls

    # Step 2: fuzzy LLM check (only if rules pass)
    tone_result = llm_judge(
        output=output,
        rubric="""Email tone should match the user's prior communication style.
        Score 5 if tone is consistent, 3 if slightly off, 1 if completely wrong.""",
        reference_examples=context.get("prior_emails", [])
    )

    return {
        "passed": rules_result["passed"] and tone_result["passed"],
        "rules": rules_result,
        "tone": tone_result,
        "feedback": rules_result["feedback"] + "\n" + tone_result.get("suggestion", "")
    }
09 / Failure Mode Map

Design for failures, not against them

Every failure mode in agent systems is predictable. These patterns were identified across all three projects. Build mitigations in at design time, not post-mortem.

Failure Mode Root Cause Mitigation Code Pattern
Premature completion Agent sees partial progress and declares done Feature list with all items "passes": false initially. Verifier checks list before declaring victory. Initializer agent creates feature_list.json
Half-baked execution Runs out of context mid-implementation, next session can't recover One feature per session. Git commits after each. Progress log entries mandatory. Coding agent CLOSING SEQUENCE
Runaway loops No iteration ceiling; agent loops on same tools Hard max_iterations on every agent. Log when limit hit and force text-only response. AgentConfig.max_iterations
Context overflow Large file contents + long history fills window Agentic search (grep) before full reads. Auto-compact at 80% token limit. smart_read() + run_with_compaction()
Context pollution Orchestrator sees all intermediate subagent steps Subagents return only final clean outputs. Orchestrator never sees raw search results. run_subagent() returns SubAgentResult
Permission creep Analysis agent accidentally has write access Separate Plan/Build configs. Deny writes in plan mode. Check before every tool call. AgentConfig.can_run_tool()
Silent failures Agent marks feature passing without testing it Mandate end-to-end testing via browser automation before any "passes": true. Coding agent WORK SEQUENCE step 8-9
Ambiguous tools Overlapping tool descriptions lead to wrong tool choice Self-contained, non-overlapping tools. Each tool has exactly one use case. GOOD_TOOLS vs BAD_TOOLS example

class AgentSafetyGuards:
    """
    Hardened agent runner with all safety guards from Anthropic's patterns.
    Wrap your agent loop with this in production.
    """
    def __init__(self, config: AgentConfig):
        self.config = config
        self.iteration_count = 0
        self.tool_call_counts: dict = {}
        self.stuck_threshold = 3  # same tool called N times in a row

    def check_iteration_limit(self) -> bool:
        self.iteration_count += 1
        if self.iteration_count > self.config.max_iterations:
            print(f"[GUARD] Max iterations ({self.config.max_iterations}) reached. Halting.")
            return False
        return True

    def check_stuck_loop(self, tool_name: str) -> bool:
        self.tool_call_counts[tool_name] = self.tool_call_counts.get(tool_name, 0) + 1

        # Reset counter if different tool called
        if tool_name != getattr(self, 'last_tool', None):
            self.consecutive_same_tool = 0
        else:
            self.consecutive_same_tool = getattr(self, 'consecutive_same_tool', 0) + 1

        self.last_tool = tool_name

        if self.consecutive_same_tool >= self.stuck_threshold:
            print(f"[GUARD] Agent appears stuck — same tool '{tool_name}' called "
                  f"{self.consecutive_same_tool}x in a row")
            return False
        return True

    def check_permission(self, tool_name: str) -> bool:
        permission = self.config.can_run_tool(tool_name)
        if permission == Permission.DENY:
            print(f"[GUARD] Blocked tool '{tool_name}' — denied in {self.config.mode} mode")
            return False
        return True

    def get_summary(self) -> str:
        return (
            f"Session: {self.iteration_count} iterations, "
            f"tool calls: {sum(self.tool_call_counts.values())}, "
            f"tools used: {list(self.tool_call_counts.keys())}"
        )
10 / Observability

Measure everything from day one

Anthropic's explicit recommendation: start with observability. Token tracking and system visibility are critical for production deployment. You cannot optimize what you cannot measure.

Production Rule

Observability is infrastructure, not a feature. Build token-level traces, tool call logs, and per-step visibility into your agent runtime before you go to production — not as an afterthought.

import time, json, uuid
from dataclasses import dataclass, field, asdict
from typing import List, Optional
from pathlib import Path

@dataclass
class ToolCallTrace:
    tool_name: str
    input: dict
    output: str
    duration_ms: float
    success: bool
    timestamp: str

@dataclass
class AgentSessionTrace:
    session_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    agent_name: str = ""
    start_time: float = field(default_factory=time.time)
    end_time: Optional[float] = None
    iterations: int = 0
    input_tokens: int = 0
    output_tokens: int = 0
    tool_calls: List[ToolCallTrace] = field(default_factory=list)
    errors: List[str] = field(default_factory=list)
    final_status: str = "running"

    def log_tool_call(self, tool_name: str, tool_input: dict,
                    result: str, duration_ms: float) -> None:
        self.tool_calls.append(ToolCallTrace(
            tool_name=tool_name,
            input=tool_input,
            output=result[:500],  # truncate large outputs in logs
            duration_ms=duration_ms,
            success=not result.startswith("ERROR"),
            timestamp=datetime.now().isoformat()
        ))

    def finish(self, status: str) -> None:
        self.end_time = time.time()
        self.final_status = status

    @property
    def duration_seconds(self) -> float:
        end = self.end_time or time.time()
        return end - self.start_time

    @property
    def total_tokens(self) -> int:
        return self.input_tokens + self.output_tokens

    def summary(self) -> str:
        tool_counts = {}
        for tc in self.tool_calls:
            tool_counts[tc.tool_name] = tool_counts.get(tc.tool_name, 0) + 1

        return (
            f"Session {self.session_id} [{self.agent_name}] — {self.final_status}\n"
            f"  Duration: {self.duration_seconds:.1f}s | Iterations: {self.iterations}\n"
            f"  Tokens: {self.total_tokens:,} (in={self.input_tokens:,}, out={self.output_tokens:,})\n"
            f"  Tool calls: {len(self.tool_calls)} — {tool_counts}\n"
            + (f"  Errors: {self.errors}\n" if self.errors else "")
        )

    def save(self, log_dir: str = "./agent-logs") -> None:
        Path(log_dir).mkdir(exist_ok=True)
        path = Path(log_dir) / f"{self.session_id}.json"
        path.write_text(json.dumps(asdict(self), indent=2))
        print(f"[trace] saved to {path}")

# Usage: wrap every agent run with tracing
def run_agent_with_tracing(
    system_prompt: str,
    message: str,
    config: AgentConfig,
    tools: list
) -> tuple[str, AgentSessionTrace]:
    trace = AgentSessionTrace(agent_name=config.name)
    try:
        result = run_agent_loop(system_prompt, message, config, tools)
        trace.finish("success")
        return result, trace
    except Exception as e:
        trace.errors.append(str(e))
        trace.finish("error")
        print(f"[trace]\n{trace.summary()}")
        raise
    finally:
        print(trace.summary())
        trace.save()

class ProductionAgentHarness:
    """
    Complete production harness integrating:
    - Two-agent pattern (initializer + coder)
    - Context management (compaction + smart reads)
    - Permission model (plan vs build)
    - Iteration limits (safety)
    - TODO planning (coherence)
    - Verification loop (rules + fuzzy)
    - Full observability (traces + token counts)
    """

    def __init__(self, workspace: str, project_desc: str):
        self.workspace = Path(workspace)
        self.project_desc = project_desc
        self.memory = AgentMemory(workspace)
        self.todos = TodoTracker(workspace)
        self.traces: List[AgentSessionTrace] = []
        self.session_num = 0
        self.initialized = False

    def initialize(self) -> None:
        print("[harness] Phase 1: Initializing project environment")
        _, trace = run_agent_with_tracing(
            system_prompt=INITIALIZER_SYSTEM_PROMPT,
            message=f"Project: {self.project_desc}\nWorkspace: {self.workspace}",
            config=BUILD_AGENT,
            tools=AGENT_TOOLS + [TODO_TOOL]
        )
        self.traces.append(trace)
        self.initialized = True

    def run_session(self) -> bool:
        """Returns True if all features now pass."""
        if not self.initialized:
            self.initialize()

        self.session_num += 1
        messages = run_with_compaction([])  # fresh context each session

        print(f"[harness] Session {self.session_num}: coding agent starting")
        _, trace = run_agent_with_tracing(
            system_prompt=CODING_AGENT_SYSTEM_PROMPT,
            message=f"Session #{self.session_num}. Workspace: {self.workspace}",
            config=BUILD_AGENT,
            tools=AGENT_TOOLS + [TODO_TOOL]
        )
        self.traces.append(trace)

        # Check completion
        features = json.loads((self.workspace / "feature_list.json").read_text())
        all_pass = all(f["passes"] for f in features["features"])

        passing = sum(1 for f in features["features"] if f["passes"])
        total = len(features["features"])
        print(f"[harness] Progress: {passing}/{total} features passing")

        return all_pass

    def run_to_completion(self, max_sessions: int = 20) -> None:
        for _ in range(max_sessions):
            complete = self.run_session()
            if complete:
                print(f"[harness] ✓ Project complete in {self.session_num} sessions")
                self.print_final_report()
                return
        print(f"[harness] Max sessions reached. See traces for status.")
        self.print_final_report()

    def print_final_report(self) -> None:
        total_tokens = sum(t.total_tokens for t in self.traces)
        total_tools  = sum(len(t.tool_calls) for t in self.traces)
        print(f"\n=== Final Report ===\n"
              f"Sessions: {self.session_num}\n"
              f"Total tokens: {total_tokens:,}\n"
              f"Total tool calls: {total_tools}\n")
        for t in self.traces:
            print(t.summary())
