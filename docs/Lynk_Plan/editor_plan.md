If you hand a lawyer an interface that exposes markdown tags, JSON, or raw XML, your product will fail immediately, regardless of how fast your Mojo compiler runs. 

To bridge the gap between a strict compiler architecture and a lawyer's requirement for visual, rich-text formatting, you must build a **Structured Rich Text Editor** acting as a hidden LSP client.

### The Architecture: A "Headless" LSP Client

You are not building a text editor; you are building a visual state machine that synchronizes with your Mojo backend via WebSockets.

#### 1. The Frontend Core: ProseMirror (Do not use anything else)
Do not attempt to build an editor using `contenteditable` or React state. You will drown in cursor management bugs.
You must use **ProseMirror** (or Meta's **Lexical**). 
* **Why it's necessary:** ProseMirror does not store HTML; it stores its own internal Document Model (a tree structure). This is critically important because it means your frontend has an AST that mirrors your Mojo compiler's AST. 
* **The WYSIWYG Illusion:** The lawyer sees a traditional word processor (bolding, headers, margins). But under the hood, ProseMirror represents that text strictly.
* **The LSP Bridge:** You will write a WebWorker that listens to ProseMirror's `Transaction` updates. When a lawyer types a sentence, the WebWorker translates that change into an LSP `textDocument/didChange` JSON-RPC payload and fires it over a WebSocket to your Mojo server.

#### 2. Rendering Diagnostics (The "Squiggly Lines")
When your Mojo compiler detects a violation (e.g., `JUR045`), it sends an LSP `PublishDiagnostics` notification back down the WebSocket.
* You will build a ProseMirror Plugin (specifically, a `DecorationSet`).
* This plugin maps the byte-offsets from the Mojo LSP payload to the visual DOM, rendering a red squiggly line under the offending legal clause.
* When the lawyer hovers over it, a tooltip displays the exact compiler message and offers a "Quick Fix" (Code Action) generated by your LangGraph agent.

#### 3. The I/O Pipeline: Ingestion and Export
Your platform must swallow `.docx` and spit out secure PDFs.
* **Ingestion (The `.docx` Parser):** Use a library like `mammoth.js` to strip the proprietary Microsoft garbage out of the uploaded `.docx` and convert the semantic structure into a clean JSON representation that initializes your ProseMirror state.
* **Export (The PDF Compiler):** When the lawyer clicks "Finalize", the frontend serializes its exact state into clean HTML/CSS. This payload is sent to your backend, where a headless engine like **WeasyPrint** (or **PrinceXML** for extreme high-end typography) renders it into an immutable PDF, signs it cryptographically, and returns it.

If you absolutely cannot force lawyers into your custom web platform, your fallback is building an **Office.js Task Pane Add-in** for MS Word. 
* **The Logic:** Word is already on their machine.
* **The Pain:** Word's API is notoriously slow. You cannot run a true real-time LSP over it without lagging the user's typing. > You are planning a system where a human types in the editor *and* a LangGraph AI agent makes asynchronous edits. If the LangGraph agent receives a prompt to "fix the indemnity clause on page 4", but it takes 15 seconds to process, and the human is actively typing on page 4 during those 15 seconds, standard state replacement will overwrite the human's keystrokes.

It allows your Mojo compiler, your LangGraph agent, and three different lawyers to edit the exact same AST simultaneously without ever locking the document or losing a single character. This is how you achieve "Google Docs" level collaboration powered by a deterministic compiler.

### The Reality Check: Do Not Fork VS Code
You cannot simply take the VS Code repository and "swap" Electron for Electrobun. VS Code contains millions of lines of code deeply hardcoded to Node.js APIs and Electron's specific Inter-Process Communication (IPC). 

Instead, you must build a **"VS Code Architecture Clone"**. You take the core concepts of VS Code (LSP, Extensions, Command Palette) and build them natively in Electrobun.

Here is the exact architectural blueprint for your Electrobun/Mojo stack.

---

### The Electrobun Architecture Blueprint

#### 1. The UI Layer (The "Webview")
Since Electrobun uses system webviews (WebKit on Mac, Edge WebView2 on Windows), your frontend must be insanely lean.
* **The Editor:** If you want a strict code-like environment, use **Monaco Editor** (the exact editor component that powers VS Code). It has native LSP support built-in. If you need rich text (bolding, real page margins), use **ProseMirror**.
* **Framework:** Use React or Svelte, compiled via Vite. 

#### 2. The Bridge (Electrobun Typed RPC)
In Electron, passing messages between the UI and the backend is a slow, serialized nightmare. Electrobun uses **Typed RPC**. 
You will define an interface in TypeScript. When the lawyer types in the Webview, it instantly fires an RPC call: `electrobun.rpc.send('didChange', documentText)`. 

#### 3. The Main Process (Bun)
This is where Electrobun shines. The Main Process runs on **Bun**, not Node.js. 
* It catches the RPC call from the Webview.
* It manages the filesystem (saving the `.docx` or `.txt`).
* **The LSP Proxy:** Bun spawns your compiled **Mojo Binary** as a child process using `Bun.spawn()`. Bun translates the RPC calls from the frontend into standard JSON-RPC strings and pipes them into the Mojo compiler's `stdin`.

#### 4. The Compiler (Mojo)
Your Mojo engine (from our previous steps) sits here. It reads the JSON-RPC from `stdin`, runs the AST validation, and spits the diagnostics (`JUR045`, `FMT101`) back out to `stdout`. Bun catches this and pushes it to the Webview.

---

### Integrating GitHub Copilot (The Interim Step)

Because you are building a custom editor, you cannot just download the VS Code Copilot extension from the marketplace. Extensions are proprietary to Microsoft's IDE.

To get "Copilot-like" quick fixes in your custom Electrobun editor before you build your LangGraph agents, you must implement the **Ghost Text Pattern**:

1.  **The Trigger:** When the user pauses typing for 400ms, the Webview sends the surrounding text context to the Bun Main Process.
2.  **The LLM API:** Bun makes a fast REST call to an LLM provider (like Anthropic, OpenAI, or a local model via Ollama) asking it to complete the legal clause.
3.  **The Rendering:** The Webview receives the suggested text. In Monaco or ProseMirror, you render this as **Inline Decorations** (greyed-out, unselectable text).
4.  **The Acceptance:** If the user presses `Tab`, your editor captures the event, deletes the ghost decoration, and inserts the actual text into the AST.

>
> If you build a platform where users can install "Custom Legal Agents" (like VS Code extensions), you face a massive security risk. A malicious agent could read the memory of the main contract, exfiltrating sensitive M&A data.

# Product & Technical Specification: "LexIDE" (Codename)

## 1. Executive Summary
LexIDE is a high-performance, neuro-symbolic text editor designed exclusively for legal professionals. It abandons legacy Electron/Node.js frameworks in favor of an **Electrobun** wrapper, utilizing **Bun** for the main process and a custom **Mojo-based Compiler** as the LSP backend. The frontend utilizes **ProseMirror** bound to **Yjs (CRDTs)** to provide a familiar WYSIWYG experience backed by a strict, compiler-enforced Abstract Syntax Tree (AST).

---

## 2. System Architecture (The Stack)

| Layer | Technology | Purpose |
| :--- | :--- | :--- |
| **OS Wrapper** | Electrobun | Replaces Electron. Provides tiny binary size (~14MB), native OS webviews (WebKit/WebView2), and strict IPC sandboxing. |
| **Main Process** | Bun | Replaces Node.js. Handles File I/O, spawns the Mojo child process, and manages the Electrobun Typed RPC bridge. |
| **Frontend UI** | React + ProseMirror | Provides the visual editor. Translates human keystrokes into state transactions. |
| **State Engine** | Yjs (CRDTs) | Replaces standard React state. Mathematically guarantees conflict-free merging of human and AI edits. |
| **Compiler / LSP** | Mojo | The deterministic engine. Reads the AST, runs the 800+ legal rules, and outputs JSON-RPC diagnostics. |

---

## 3. Data Flow & The LSP Bridge

The core loop of the application must execute in under 16 milliseconds to maintain a 60FPS typing experience while running complex legal linting.

1.  **Keystroke (Frontend):** Lawyer types "Company".
2.  **State Update (Yjs):** ProseMirror captures the transaction. Yjs updates the shared document vector.
3.  **RPC Trigger (Electrobun):** The frontend fires `electrobun.rpc.send('didChange', { uri, version, changes })`.
4.  **Process Handoff (Bun):** The Bun main process catches the RPC and writes it to the `stdin` of the Mojo compiler process as a JSON-RPC 2.0 payload.
5.  **Compilation (Mojo):** The Mojo engine does a partial AST update, evaluates the rules, and detects a formatting violation (`LEX005: lowercase defined term`). It writes the diagnostic to `stdout`.
6.  **Rendering (Frontend):** Bun pipes `stdout` back over the RPC bridge. ProseMirror maps the byte-offsets from Mojo to the DOM, rendering a red squiggly line.

## 4. Core Editor Features (Frontend)

### 4.1 Structured Rich Text (The ProseMirror Schema)
The editor will not allow raw HTML injection. The ProseMirror Schema must be strictly defined to match the Mojo AST.
* **Allowed Nodes:** `Doc`, `Paragraph`, `Heading (1-4)`, `DefinitionBlock`, `SignatureLine`, `List (Ordered/Unordered)`.
* **Allowed Marks:** `Bold`, `Italic`, `Underline`, `CrossReference`, `InvisibleHash`.

### 4.2 Diagnostic UI
* **Squigglies:** Different colors for different severities (Red = Statutory Error, Yellow = Ambiguity, Blue = Formatting).
* **Hover Tooltips:** When hovering over a diagnostic, the UI displays the exact Mojo compiler output (e.g., *"ForeignSeatTwoIndianParties: Two Indian entities cannot choose a foreign seat..."*).
* **Code Actions:** A lightbulb icon appears offering "Quick Fixes" (e.g., "Change seat to Mumbai").

## 5. AI Integration Pipeline

### Phase 1: Local Copilot (The Ghost Text Engine)
Before LangGraph agents are integrated, the editor needs immediate, context-aware autocompletion.
* **Trigger:** 400ms debounce on typing pause.
* **Context Gathering:** Bun extracts the previous 2,000 tokens and the next 500 tokens from the Yjs state.
* **Inference:** Bun makes a fast API call (e.g., Anthropic Claude 3.5 Sonnet / local Llama 3) with a strict system prompt: *"Complete the legal clause. Output raw text only."*
* **Rendering:** ProseMirror renders the response as an inline, non-selectable `Decoration`.
* **Commit:** Pressing `Tab` fires a Yjs transaction to convert the decoration into actual text nodes.

### Phase 2: LangGraph Agent Injection via OOPIF
* **The Sandbox:** Custom legal agents (e.g., "M&A Diligence Agent") do not run in the main thread. Electrobun spawns an **Out-of-Process iframe (OOPIF)**.
* **The Neuro-Symbolic Loop:** 1. User clicks "Rewrite this section for Employer Favorability."
    2. LangGraph agent reads the section and streams JSON modifications.
    3. Mojo compiler intercepts the stream, checking it against Indian Labor Law.
    4. If it fails, Mojo rejects the agent's edit and forces the agent to try again.
    5. Once Mojo approves, the text is injected into the ProseMirror Yjs state.

## 6. Document I/O Pipeline

### Ingestion (`.docx` $\rightarrow$ AST)
1.  User drags and drops `MSA_Draft_v4.docx`.
2.  Bun intercepts the file, unzips the OOXML, and streams `document.xml` to the Mojo parser.
3.  Mojo creates the Legal Syntax Tree, translates it into ProseMirror JSON, and initializes the Yjs state.

### Export (AST $\rightarrow$ PDF/A-3)
1.  User clicks "Finalize & Sign".
2.  Frontend locks the Yjs state to `ReadOnly`.
3.  ProseMirror serializes the document into a headless HTML string + CSS Print Stylesheet.
4.  Bun passes the HTML to **WeasyPrint** (or an equivalent native renderer).
5.  Bun uses a cryptography library to append a SHA-256 hash of the final JSON state into the metadata of the PDF (chain of custody).

>
> If the user types *"the company"* and your Mojo engine's auto-formatter instantly capitalizes it to *"the Company"*, the standard browser DOM will reset the user's cursor to the beginning of the word, breaking their typing flow.

When Mojo sends a "Quick Fix" or "Auto-Format" command back over the LSP bridge, it must include the exact byte-offset of the change. Your frontend WebWorker must intercept this, calculate the delta, and programmatically shift the ProseMirror `Selection` object forward by exactly the number of characters modified, *before* the browser paints the next frame.
